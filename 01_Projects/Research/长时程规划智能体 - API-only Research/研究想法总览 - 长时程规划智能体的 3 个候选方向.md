## 总体方向
最初的 idea report 提出了 3 个值得优先考虑的方向，目标都是在 **training-free、API-only** 条件下改进长时程规划 agent 的恢复能力与稳定性：
- **Plan Splicing**
- **Counterfactual Failure Attribution**
- **Triggered Replanning via Hazard Scores**

## 3 个方向的关系
这 3 个 idea 可以看成同一条主线上的不同问题：

### 1. 什么时候该重规划
- **Triggered Replanning via Hazard Scores**
- 关注的是 replanning trigger：什么时候值得打断当前执行、启动 repair / replan

### 2. 失败后应该修哪里
- **Counterfactual Failure Attribution**
- 关注的是 causal diagnosis：真正导致失败的根因步骤在哪里

### 3. 修的时候应该怎么修
- **Plan Splicing**
- 关注的是 repair policy：应该局部修补，还是整体重规划

## 当前进展：主线已经收敛
根据新增的 `FINAL_PROPOSAL.md` 和 `EXPERIMENT_PLAN.md`，当前研究已经明显从“并列 idea 候选”收敛到：

### 当前主推方案：Plan Splicing
Plan Splicing 已经从一个 brainstorming idea，推进成了带有明确方法定义、差异化定位和实验验证路线的 proposal。

它的核心优势在于：
- 问题定义明确
- 和 full replanning、suffix replanning 的 baseline 对比自然
- 指标可以设计得很清晰
- API-only，不需要训练

## 我对这 3 个方向的当前判断
### 最适合优先推进的
- **Plan Splicing**
- 原因：方法已经成型，实验计划已经明确，最接近可以直接开做

### 最适合做后续增强模块的
- **Counterfactual Failure Attribution**
- 原因：未来可以作为 failure analyzer 的增强版本，帮助更准确定位 invalid region

### 最适合做低成本平行实验的
- **Triggered Replanning via Hazard Scores**
- 原因：成本低、验证快，即使是负结果也有意义

## 当前结论
所以这篇笔记现在不应再被理解为“3 个平级候选都一样重要”，而应该理解为：
- 候选池里已经筛出主线：**Plan Splicing**
- 其余两个方向仍有价值，但更像后续扩展或平行验证方向
