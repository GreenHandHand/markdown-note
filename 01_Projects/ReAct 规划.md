# 规划

> [!definition|目标] 将规划内容整理为结构化信息，并构建一个能够基于现有信息持续整理、分析与探索的智能 Agent。

## 主要结构

为尽可能保证 Agent 的自主性，这里采用 **原子卡片 + 轻量连接** 的方式组织信息。系统不预先规定复杂图结构，而是先设计一个最小信息单元，使 Agent 能围绕已有内容不断生成、整理和扩展。

> [!definition|原子卡片]
> 原子卡片是系统中的最小信息单元，也是 Agent 的基本操作对象。每个原子卡片包含：
> - `title`：卡片标题
> - `desc`：一句简短描述，用于全局浏览
> - `content`：卡片完整内容
> - `link`：与其他卡片的连接信息
> - `type`：卡片类型，引用自类型池
>
> 此外还可以包含时间、来源、状态等 metadata。

> [!note]
> `title` 与 `type` 共同构成卡片的逻辑唯一约束。系统不允许创建名称与类型完全相同的卡片；若出现冲突，系统会提示 Agent，并展示已有卡片，由 Agent 决定如何处理。

> [!note]
> 对 Agent 而言，它不需要知道当前结构最终呈现为图、树还是其他形式。Agent 只需要对卡片进行增删改查、建立连接、发现缺失并继续探索。

## type

> [!definition|type] 卡片类型
> **`type` 不是固定枚举，而是 Agent 在长期整理过程中逐步收敛出的轻量概念集合。**

系统维护一个类型池，所有卡片的 `type` 都引用自该池中的类型。一个类型包含：

- `type_name`：类型名称
- `desc`：类型说明

> [!note]
> type 的作用是帮助 Agent 在长期整理中形成稳定、可复用的概念，而不是提前限制 Agent 的思考方式。

> [!note]
> 仅当现有 type 无法准确覆盖某类卡片，且该语义具有复用价值时，才创建新的 type。系统应尽量保持 type 池简洁。

> [!tip]
> 系统内置一个 builtin type：`untyped`，用于表示当前暂时无法可靠归类、或因类型变更而失去原有语义匹配的卡片。

> [!note]
> 系统提供 `update_type` 操作，用于修改已有 type 的定义。修改后触发 hook，由 subagent 检查受影响的卡片是否仍然符合新的 type 语义；不符合的卡片会被重新标记为 `untyped`。

## untyped 与 reminder

> [!definition|definition] untyped
> `untyped` 表示待重新整理的卡片类型。

> [!note]
> 只要系统中存在 `untyped` 卡片，就会在后续对话中持续注入 reminder，提醒 Agent 当前仍有待整理内容。系统不强制立即处理，但这些提醒会一直保留，直到对应卡片被重新整理。

## Harness 与上下文

系统以 Agent 为中心，不替 Agent 做过多决策，而是为 Agent 提供进行判断所需的信息与访问能力。

当前阶段的上下文组织如下：
- 种子卡片提供 `content` 全文
- 全部卡片提供 `title + desc` 的全局摘要
- 其他卡片内容按需展开

> [!note]
> 这种设计的核心是：系统提供全局视图与访问能力，Agent 自己决定关注哪些卡片、展开哪些内容。

## 种子卡片

种子卡片是一次分析的起点，来源有两种：

- 用户主动选中的卡片
- 新创建的卡片

系统围绕种子卡片组织上下文，供 Agent 在局部范围内开展分析与探索。

## 渐进式展开

为了控制上下文规模，系统采用渐进式展开机制。当前仅提供一个最小工具：

- `load_card(card_id)`：读取某张卡片的全文内容

> [!note]
> Agent 先通过全局摘要判断潜在相关卡片，再使用 `load_card` 深入查看。系统不预先替 Agent 构造复杂局部子图，而是让 Agent 通过选择性读取逐步建立自己的局部上下文。

## gap

在本系统中，gap 不只是简单的信息缺失，而是指：

> **多个已有卡片共同暗示某个本应存在、但尚未被显式写出的邻居卡片。**

这个缺失的邻居卡片可以是：

- 问题
- 解释
- 总结
- 其他补充现有结构的节点

> [!note]
> gap 的识别主要由 Agent 基于语义进行判断。系统负责提供种子、全局摘要与按需访问能力，Agent 负责分析已有卡片之间的潜在关联、结构缺口与新的探索方向。
