# 规划

> [!definition|目标]
> 将知识组织为以原子卡片为核心的持续生长结构，并构建一个能够围绕 cards 自主整理、检索、分析、验证与沉淀的 Agent 原型。

## 主要设计

### 原子卡片

> [!definition|原子卡片]
> 原子卡片是系统中的最小知识单元，也是 Agent 的基本操作对象。每张卡片包含：
> - `title`：卡片标题
> - `desc`：一句简短描述，用于注入、检索与全局浏览
> - `content`：卡片完整内容（markdown）
> - `type`：卡片类型，引用自类型池
> - `links`：从当前卡片发出的连接信息
> - `metadata`：可选附加信息，如来源、状态或其他轻量字段
> - 维护信息：
> 	- `card_id`：用于系统管理，与 Agent 功能无关。
> 	- `created_at` / `updated_at`：系统维护的时间信息

> [!note] `desc` 不是可有可无的附属字段。创建或更新卡片时，Agent 需要同步维护 `desc`，保证其能够持续作为局部工作集的稳定摘要。

> [!tip] 原子卡片把知识沉淀、检索入口和后续整理统一到同一个最小对象上。系统不预先规定复杂图结构，而是让结构从大量稳定的小单元中逐步长出来。

### link

> [!definition|link]
> `link` 是带注释的有向边。每条边至少包含：
> - `target_card_id`：目标卡片
> - `note`：一段简短说明，解释这条连接为何成立

> [!note]
> 当前阶段的 link 不引入独立的关系类型枚举，也不把边建模为单独实体。系统只要求：
> - 连接有方向
> - 连接有简短说明
> - 连接可以被 Agent 读取、补充和更新

> [!tip] 有向边让 Agent 可以表达“从哪里指向哪里”，短注释让连接本身也能携带少量语义。这样既保持最小结构，又不会把关系压缩成完全无解释的裸边。

### type

> [!definition|type]
> `type` 不是固定枚举，而是 Agent 在长期整理过程中逐步收敛出的轻量概念集合。
>
> 一个类型至少包含：
> - `type_name`：类型名称
> - `desc`：类型说明

> [!note] untype
> 系统内置一个 builtin type：`untyped`，用于表示当前暂时无法可靠归类、或因类型变更而失去原有语义匹配的卡片。
> 当前阶段允许 Agent 直接创建与修改 type。修改 type 后，系统应立即批量复检受影响卡片；若某张卡片已不再符合新的 type 语义，则重新标记为 `untyped`。
>
> > [!note] reminder
> > 只要系统中存在 `untyped` 卡片，就会在后续上下文中持续注入汇总提醒，例如“当前仍有 N 张 untyped 卡片待整理”。当前阶段不在 reminder 中直接列出大量卡片，也不让 reminder 阻塞主线工作。
>
> > [!tip] `untyped` 让系统可以先接纳尚未整理完成的知识，再逐步收敛结构。汇总提醒保证 Agent 不会长期遗忘整理债务，同时避免反复向上下文注入噪声。

> [!tip] type 的目标不是提前限制思考，而是让 Agent 在长期整理中逐步形成可复用的概念骨架。允许直接演化 type，可以保持系统的自主性；立即复检则用来防止类型漂移失控。

### Harness、工作集与渐进式展开

> [!definition|Harness]
> 系统以 Agent 为中心，但不替 Agent 做完整判断。系统负责：
> - 维护会话与运行时状态
> - 构造当前工作集
> - 注册 cards 访问能力
> - 注入必要 reminder
> - 提供完整事件流给前端展示

> [!tip] 这里的关键不是“系统替 Agent 决定答案”，而是“系统稳定地准备好 Agent 做判断所需的上下文和接口”。

> [!definition|工作集]
> 工作集是当前轮直接进入 Agent 上下文的 cards 摘要集合。

> [!note] 工作集构造方式
> 当用户选中某张 card 时，工作集默认包含：
> - 当前选中的 card
> - 该 card 的一跳邻居
> - 最近修改的 cards
> - 然后统一去重
>
> 邻居跳数不是固定语义，而是 service 侧可调超参数。当前阶段默认按一跳理解。

> [!note]
> 当工作集合为空时，也就是**用户没有选中卡片、没有最近修改的卡片**时，注入提示词，要求：
> - service 提供语义检索工具
> - Agent 在完整首轮中自行生成检索 query 检索需要的卡片信息

> [!note] 新创建的 card 在当前轮立即回流到工作集中，避免 Agent 创建后又立刻失去对该结果的上下文感知。

> [!warning] 限制
> 当前阶段不强制工作集的硬性数量或 token 上限，而是依赖 `desc` 足够短来控制规模。后续如出现上下文膨胀，再增加预算控制、裁剪措施。

> [!tip] 工作集的目标是替代“全局目录注入”。系统只给 Agent 当前最值得关注的一小片知识表面，剩余内容通过检索和按需展开进入上下文。

### Cards 访问能力

> [!definition|Cards 访问能力]
> 当前阶段，cards 相关 tools 由 `service` 拥有并注册给 Agent。最小能力包括：
> - `search_cards(...)`：基于 query 检索 cards 摘要
> - `load_card(card_id)`：按 `card_id` 读取完整内容
> - `create_card(...)`：创建普通 card
> - `update_card(...)`：更新已有 card，包括内容、type 与 links

> [!note]
> Agent 并不直接持有 `cards` 内部实现；它通过 service 注册的能力来访问 cards。对 Agent 来说，cards 是一个可探索、可读取、可整理的知识表面。

> [!note|用意]
> 让 `cards` 保持纯领域模块，让 `service` 负责把它包装成 Agent 可用接口，可以同时满足清晰边界与 Agent 自主性。

### terminal

> [!definition|terminal]
> 底层将卡片存储为文件系统，使得 Agent 可以通过编写脚本、执行终端命令来达到对卡片更强大的操纵能力。

## 用户与 Agent 的分工

> [!definition|默认操作者]
> 系统由 Agent 主导整理与写入，用户负责监督与纠错、主导方向：
> - Agent 可以直接创建和更新 cards
> - Agent 可以直接创建和修改 type
> - Agent 可以补充和修改 links

> [!note]
> 用户不是被动旁观者。用户可以随时：
> - 查看 Agent 当前推进
> - 纠偏
> - 修改已有知识
> - 覆盖 Agent 的整理结果
> - 继续追加指令改变方向

> [!tip] 这个系统的默认形态是“Agent 主导，用户可纠偏”，而不是“所有知识写入都必须先经用户批准”。只有这样，原型才能真正体现 Agent 工作台的连续推进能力。

### 过程可见性与事件流

> [!definition|事件流]
> 当前阶段的过程可见性主要由后端事件流支撑。后端需要提供完整的流式事件，至少包括：
> - 文本增量
> - 思考片段
> - 工具调用
> - 工具结果
> - 关键状态变更

> [!note] 当前阶段的过程状态主要由前端对话框展示。也就是说，后端此时更需要提供完整事件，而不是单独承担一个复杂的工作台状态模型。

> [!tip] 用户需要看见 Agent 正在做什么，但原型阶段不必一次性做完整的专用工作台状态系统。先把事件流做完整，前端就有足够材料来展示过程。
