# 评测

本笔记用于整理 CardAgent 的评测框架。

> [!note] CardAgent 的评测对象不是某一次回答是否正确，也不是最终图谱是否等于某个唯一标准答案，而是系统是否能够帮助 Agent 更好地理解、复用、整理和扩展 card graph。

由于开放式知识组织任务通常不存在严格的 gold graph，评测需要从**隐含知识结构**的角度出发：给定一个 card graph，系统背后存在某种理想的知识组织结构，包括概念边界、语义关系、类型归属、局部子图组织方式以及后续扩展路径。评测的核心不是直接还原这一隐结构，而是通过一组可观测任务间接判断当前图谱和 Agent 行为是否更接近这种高质量组织状态。

## 形式化定义

设当前 canvas 中的 card graph 为：

$$
G = (V, E, T, M)
$$

其中，$V$ 表示 card 集合，$E$ 表示有向 link 集合，$T$ 表示 type 集合，$M$ 表示 card 与 link 上的辅助信息，包括 `desc`、`note`、metadata 等。

设高质量知识组织结构为隐变量：

$$
z
$$

$z$ 不要求被完整显式标注，也不要求构造完整 gold graph，而是表示一个好的 card graph 应当满足的组织规律，例如相关知识可被检索、相近概念不过度重复、类型体系稳定、link 具有明确语义、新增知识能够接入已有结构。

基于 $z$，可以构造一组 probe tasks：

$$
Q(z) = \{q_1, q_2, \ldots, q_m\}
$$

每个任务 $q_i$ 用于评估当前图谱在某一类 Agent 工作场景下的支持能力。

> [!note] 评测对象
> 这里评测的不是 Agent 是否生成了唯一正确答案，而是当前 card graph 是否能支持 Agent 更好地检索、复用、整理和扩展知识。

## 两种评测模式

CardAgent 的评测需要同时覆盖两种模式。

第一种是**长程任务评测**。Agent 从一个初始状态出发，持续执行多步检索、读取、写入和整理操作，并在过程中记录每一步工具调用、图谱变更和中间状态。这种方式更贴近真实使用场景，适合观察 Agent 是否能在长程任务中保持连续性、复用已有知识并减少结构债务。

第二种是**给定图谱的局部任务评测**。系统从一个人工构造或真实截取的 seeded graph 出发，要求 Agent 完成一个明确的 probe task。这种方式更利于控制变量和设计指标，例如评估 Agent 是否能发现重复 card、补充缺失 link、整理 type 或将新增知识接入已有结构。

> [!tip] 两种模式的关系
> 长程任务评测更接近真实使用，但变量复杂；给定图谱的 probe task 更容易控制变量，但场景相对人工。前者用于验证系统整体价值，后者用于分析具体能力来源。两者需要结合使用。

## 领域定义和理由

在领域定义上，CardAgent 的评测不应首先关注**答案正确率**，而应关注 Agent 在 card workspace 中的**知识整理行为**。具体而言，评测应围绕以下能力展开：

- 已有知识检索
- 复用而非重复创建
- 结构整理
- 跨轮连续推进
- 知识扩展质量

这些能力对应 CardAgent 的核心目标：让 card graph 成为 Agent 可持续使用的知识工作面，而不是一次性生成结果的容器。因此，评测指标应尽量从 graph 状态、git diff、tool trajectory、embedding similarity 等可观测信号中自动计算，避免过早依赖复杂人工标注或大规模 LLM-as-judge。

> [!note] 设计理由
> 开放式知识组织任务通常没有唯一标准答案。相比构造完整 gold graph，更合理的做法是构造一组 probe tasks，观察 Agent 是否能在固定场景下表现出更好的检索、复用、整理和扩展行为。

## 每个任务的通用定义

每个 probe task 采用统一定义：

$$
q_i = (a_i, G_0, goal_i, d_i^{primary}, D_i^{secondary}, D_i^{excluded})
$$

其中：

- $a_i$ 表示任务所属的应用场景；
- $G_0$ 表示初始 seeded graph；
- $goal_i$ 表示给 Agent 的任务目标；
- $d_i^{primary}$ 表示主测维度；
- $D_i^{secondary}$ 表示次测维度；
- $D_i^{excluded}$ 表示本任务明确不评价的维度。

这样的定义可以避免一个任务同时测试过多能力，导致评测结果不可解释。对于每个任务，应先说明其应用场景、期望 Agent 行为和典型失败模式，再定义可观测指标。

通过这种方式，可以判断具体能力到底改善了什么。

> [!example] 任务解释边界
> 如果一个任务的主测维度是 `reuse_vs_recreate`，那么它可以次要观察 `retrieval_support`，但不应同时评价 `knowledge_extension_quality`。否则最终分数会混合多个能力来源，难以解释。

## 指标设计

### B. 重复节点

Agent 能否在当前 card graph 中发现语义重复节点，并以安全、可解释、非破坏性的方式输出重复候选，避免继续制造重复结构。

> [!note] Dataset
> WDC Products 的原始数据是 product offer。每条记录表示一个电商网页中的商品报价或商品条目。Hugging Face 上的 `wdc/products-2017` 字段包括：`id`、`category`、`cluster_id`、`brand`、`title`、`description`、`price`、`specTableContent`。其中 `cluster_id` 表示该 offer 属于哪个产品簇。

数据集被处理为：

```
class BenchmarkCase:
  seed_graph = [cards, links, types],
  labels: [
	  duplicate_pairs, 
	  duplicate_clusters,
      hard_negative_pairs,
      expected_retrieval,
      expected_orphans,
      expected_link_targets,
  ],
  tasks: [
    task_id,
    input,
    expected_output_shape,
    scoring_scope,
  ]
```

#### Duplicate Judgement

- 任务：给定一个查询或用户需求，Agent 是否能从当前 card graph 中找到相关已有 cards。
- 预处理：预先构建图结构
- 输入：
	- query: 用户查询
- 输出：
	- `path`: Agent 的行为路径
	- `is_duplicate`：二分类指标，是否重复
	- `duplicate_card_id`：重复卡片 id
	- `Optional[answer]`: Agent 对重复的解释，以及之后的处理方式
- 评价方式：
	- 保证 path 尽量短的同时，二分类正确。正确时得分为 1/path 长度，错误时得分为 0。

#### Duplicate Discovery

- 任务：给定一个包含重复节点和 hard negatives 的 graph，Agent 是否能主动发现 duplicate nodes。
- 预处理：预先构建图结构
- 输入：
	- `task_prompt`：要求 Agent 检查重复节点。
- 输出：
	- `duplicate_pair`: 重复节点对
	- `desc`：每个重复节点对的解释与处理方式
- 评价方式：
	- 一些多分类指标，判断重复节点对是否正确
	- LLM-as-judge 对于每个重复节点的处理方式判断是否合理
