# 智能探索 Agent · 需求分析

> [!definition|目标]
> 构建一个以原子卡片为核心的智能探索系统。用户围绕已有 cards 进行持续整理、检索、分析与沉淀，Agent 主导结构化工作，用户负责监督、纠偏与方向控制。

## 1. 产品定位

> [!summary] 这是一个由 Agent 主导、围绕原子卡片持续整理与探索的知识工作台原型。

> [!note] 系统的目标不是一次性生成答案，而是帮助用户把零散信息逐步沉淀为一个可生长、可检索、可回溯、可继续探索的知识结构。

## 2. 用户如何使用

> [!definition|默认使用方式]
> 用户通过 **选择 card + 对话下达目标** 的方式驱动系统。

典型使用方式包括：
- 选中一张 card，要求 Agent 继续整理、扩展或分析 gap
- 不选中 card，要求 Agent 先检索并建立初始切入点
- 新增一张 card 后，让 Agent 将其接入现有结构
- 在过程中随时纠偏、修改、覆盖 Agent 的结果

> [!tip] 系统默认是 **Agent 主导，用户可纠偏**，而不是每一步都由用户审批。

## 3. 系统如何回应

> [!definition|单轮响应形态]
> 一轮工作的理想过程如下：

1. 用户输入当前目标  
2. 系统构造工作集并注入上下文  
3. Agent 自主检索、读取、分析、整理  
4. 系统流式展示过程事件  
5. Agent 输出本轮结果并停止

Agent 在单轮中可以：
- 检索 cards
- 读取全文
- 创建或更新 cards
- 补充 links
- 创建或修改 type
- 发现 gap 并生成新的问题、解释或总结卡片

> [!note]
> 一轮结束时，Agent 应明确说明：
> - 本轮做了什么
> - 修改了哪些对象
> - 发现了什么新的结构或 gap

## 4. MVP 核心能力

> [!check] 当前版本必须支持
> - 原子卡片的创建、更新、检索、加载
> - 轻量 links 的维护
> - type pool 与 `untyped`
> - type 修改后的受影响卡片复检
> - 基于种子 card 的工作集构造
> - `load_card` 驱动的渐进式展开
> - 用户可中途纠偏
> - 完整事件流展示 Agent 过程

## 5. 当前边界

> [!warning] 当前版本不做
> - 不做完整知识图谱推理系统
> - 不做全局最优自动整理
> - 不做复杂审批流
> - 不做重型工作流面板
> - 不做复杂 token 预算控制
> - 不做强形式验证系统

> [!note] 当前阶段只要求系统在局部工作集中稳定推进，而不是一次性整理整个知识库。

## 6. 成功标准

> [!success] MVP 成立的判断标准
> 如果系统满足以下条件，就说明项目核心边界成立：

- 用户可以稳定地通过“选中 card + 给出目标”的方式使用系统
- Agent 能围绕已有结构持续推进，而不只是回答问题
- 用户能看懂 Agent 在做什么，并可随时纠偏
- 多轮对话后，结构比初始状态更清晰
- 用户能明显感受到系统在帮助自己理清思路

> [!summary] 核心判断
> 这个系统是否成立，不取决于它是否“智能得很强”，而取决于它是否真的能帮助用户把已有信息持续整理成更清晰的结构。
